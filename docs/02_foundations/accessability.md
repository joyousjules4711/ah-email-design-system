> **Applies to**
>
> - All transactional email templates
> - All reusable components
> - German and English versions

# Accessibility

Accessibility is a fundamental principle of the Akademikerhilfe Email Design System.

Every transactional email should be readable, understandable and usable by as many people as possible, regardless of device, ability or email client.

Accessibility is considered throughout the entire design and development process—not as an afterthought.

---

# Purpose

Accessible emails improve usability for everyone.

This includes users who

- use screen readers
- have visual impairments
- have limited motor abilities
- use mobile devices
- read emails under poor lighting conditions
- rely on larger text sizes

Accessibility also improves readability for all recipients.

---

# Accessibility Principles

Every email should be

- perceivable
- operable
- understandable
- robust

These principles are based on the Web Content Accessibility Guidelines (WCAG) whenever applicable to email clients.

---

# Colour and Contrast

Colour must never be the only way to communicate information.

Always combine colour with

- icons
- labels
- headings
- descriptive text

---

## Contrast

Text should maintain sufficient contrast against its background.

Avoid light grey text on white backgrounds.

Primary content should meet WCAG AA contrast recommendations whenever technically possible.

---

# Typography

Body text should remain easy to read.

Recommended minimum sizes

| Element | Size |
|----------|------|
| Body text | 16 px |
| Secondary text | 14 px |
| Headings | 28–36 px |
| Buttons | 16 px |

Avoid very small text.

---

# Line Length

Keep paragraphs reasonably short.

Avoid very wide text blocks.

Recommended email width

660 px

---

# Line Spacing

Recommended line height

1.5–1.7

Generous spacing improves readability.

---

# Headings

Use a logical heading hierarchy.

Every email should contain one H1.

Subsections should use H2 where appropriate.

Do not skip heading levels.

---

# Plain Language

Use simple language.

Avoid unnecessary technical terminology.

Use legal terminology only where required.

Keep sentences concise.

---

# Links

Links should clearly describe their destination.

Good

> Open the Online Portal

Avoid

> Click here

Links should remain understandable without surrounding text.

---

# Buttons

Buttons should

- contain action-oriented text
- be large enough to tap on mobile devices
- have sufficient spacing
- provide clear contrast

Recommended minimum height

44 px

---

# Images

Images should support the content—not replace it.

Do not place essential information inside images.

Decorative images should not contain meaningful text.

---

# Alternative Text

Meaningful images must include descriptive alternative text.

Example

```
Akademikerhilfe logo
```

Avoid

```
image
```

or

```
logo
```

---

# Tables

Use tables only where appropriate.

Payment schedules and financial information are suitable examples.

Avoid complex nested tables unless required for email layout.

Where layout tables are required, they should remain as simple as possible.

---

# Responsive Design

Emails should remain readable on

- desktop
- tablets
- smartphones

Content should reflow naturally without horizontal scrolling.

Buttons should remain easy to tap.

---

# Mobile Experience

Users should not need to zoom.

Important information should appear near the beginning of the email.

Primary actions should remain easily accessible.

---

# Screen Readers

Write meaningful headings.

Use descriptive link text.

Avoid excessive formatting.

Avoid using emojis as the only source of meaning.

---

# Icons

Icons should support understanding.

Icons must never replace descriptive text.

---

# Animations

Avoid animations that may distract users.

If animations are used, they should never contain essential information.

---

# Attachments

Always explain

- what the attachment contains
- why it is included
- what the recipient should do with it

---

# Error Prevention

Important actions should always be clearly explained.

Recipients should understand

- what is expected
- why it is required
- what happens next

---

# Email Testing

Accessibility should be verified during template development.

Recommended checks include

- colour contrast
- keyboard readability
- mobile readability
- screen reader compatibility
- Outlook rendering
- Gmail rendering

---

# Accessibility Checklist

Before publishing a template, verify that

- all text is readable
- headings follow a logical order
- colours meet contrast requirements
- buttons are easy to tap
- links describe their destination
- images contain alternative text
- important information is not conveyed by colour alone
- the email remains readable on mobile devices

---

# Guiding Principle

Accessibility is not a feature.

It is a quality standard.

Every improvement in accessibility improves the experience for all recipients.
